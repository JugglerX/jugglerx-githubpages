Stereotype Threat

Apr 5th 2015 / Cultural Blog

Stereotype threat is an academic idea which argues people in minorities will take on the negative traits of a stereotype in certain situations.

Of particular concern is how sterotype threat affects the academic and career performance of racial miniorities and women.

"Stereotype threat is a potential contributing factor to long-standing racial and gender gaps in academic performance."

I don't know as I have ever been affected by stereotype threat myself. However I certainly have moments of self doubt, anxiety etc.

Relationships based on trust and respect. Employment based on merit and performance.

This is a big issue, no doubt. And race, nor gender should hinder the opportunities for an individual. Relationships based on trust and respect. Employment based on merit and performance. These are standards to which we should strive.

Stereotypes exist, and we box people into them all the time. When I find myself sterotyping someone, especially negatively I remind myself "I don't know this person at all. They could be really interesting, they could be amazing. It takes time to know who someone really is."

Variable Scope

Apr 5th 2015 / Technical Blog

Ruby has the following variables. Each variable type has a different variable scope.


  $global_var = 10
  CONSTANT = 10
  @@class_var = 10
  @instance_var = 10
  local_var = 10

Local Variables

local variables have the most limited scope. They are limited to the method or loop in which they are declared. The location and depth of the local variable will determine inheritance. 


def local
    (1..5).each do |x|
      var = x
    end
    p var #=> undefined local variable or method `var' for main:Object
end

If we declare the variable var above the loop it will be available to the entire method


def local
    var = 0
    (1..5).each do |x|
      var = x
    end
    p var #=> 5
end

Class Variables

A Class variable is available to all instances of the class.


class Ticket
  def initialize(price)
    @@price = price
  end
  def check_ticket_price
    p @@price
  end
end
ticket_one = Ticket.new(15)
ticket_one.check_ticket_price #=> 15
ticket_two = Ticket.new(10)
ticket_one.check_ticket_price #=> 10

ticket_two.check_ticket_price #=> 10

The downside to a class variable is that if any instance of the Ticket class updates the variable, it is updated everywhere. Not very useful for tickets with individual pricing is it?

Class Instance Variables

A Class instance variable is unique to each instance of a class. This avoids the problem above.


class Ticket
  def initialize(price)
    @price = price
  end
  def check_ticket_price
    p @price
  end
end
ticket_one = Ticket.new(15)
ticket_one.check_ticket_price #=> 15
ticket_two = Ticket.new(10)
ticket_one.check_ticket_price #=> 10

ticket_two.check_ticket_price #=> 15

Now we can spawn as many individual tickets as we like, each with there own price.

Global Variables

Global variables are accessible everywhere, no matter where they are declared.


def local
    (1..5).each do |x|
      $var = x
    end
end

p $var #=> 5

Constants are the same as the $global_variable except when you try to change them they give you a warning.

Classes

March 29th 2015 / Technical Blog

Classes are like groups. Classes allow you to define methods and variables inside of them and reuse them across multiple objects.

Typically you would write a new class like so.


  class Person
    def name
      name = "Rob"
    end
  end

You are then able to spawn new instances of that class object. These new objects have access to the methods and variables defined in the Class.


  # Create new object which is an instance of the Person Class
  new_customer = Person.new
  p new_customer.name #=> "Rob"

So far, pretty easy right? You should be able to see the potenial of classes for everyday applications.

Let's add the ability to change the name variable and see what happens


class Person
  @first_name = ""
  def first_name
    @first_name
  end
  def change_first_name(name)
    @first_name = name
  end
end

new_customer = Person.new
p new_customer.first_name #=> nil
new_customer.change_name("Bob")
p new_customer.first_name #=> "Bob"

Now we've defined two methods in our class. One to display the first_name and the other to change it. We've also introduced a new type of variable @first_name which is called a class instance variable. The local variable we were using did not have enough scope to talk between the seperate methods #first_name and #change_first_name.

There is also a third variable called a class variable which looks like @@first_name. Class variables are inherited by new class instances and class instances of descendant classes. But if you change a class variable anywhere it will change everywhere, which may not be desirable.

Other important concepts that go with Classes are attributes, understanding class variables vs class instance variable scope and inheritence, understanding context and self and how that applies to class objects and the main class.

Regulation

March 22nd 2015 / Cultural Blog

This week we've been asked to research and provide our opinion on an issue facing the tech industry. I feel regulation is the largest looming threat to technology.

Most would agree that regulation stifles innovation, but that some degree of regulation is needed to stop "bad things happening".

My concern is that more often than not, regulation is actually used as a tool by the government to protect existing political or private interests.

Relevant examples at the moment include AirBNB, Uber, Bitcoin and the erosion of privacy on Facebook and Google.

Uber: An entrenched cab union and lucrative licensing fees for local and state governments conspire to stop competition under the pretense of public safety. Uber is a cheaper, more efficient and relatively safe way for people to travel. (just as safe as cabs)

AirBNB People choose AirBNB because it's a cheaper and more human experience - It's the opposite of a hotel. But the hotel lobby and government think it should operate MORE like a hotel. Maybe one day AirBNB will just be another hotel booking aggregator.

In both scenarios we see great new services which are making travel and accommodation cheaper and more flexible for hundreds of thousands of people.

But they risk being regulated to the point of uncompetitiveness - restricting the innovations that made them so popular in the first place seems pointless and short sighted to me. Especially because the reasons for the regulation are often dubious and flimsy.

Ruby .map Method

March 22nd 2015 / Technical Blog

Ruby provides various methods to iterate over arrays and hashes. The array and hash classes both come with the #each method which you are probably familiar with.

But there are several other powerful methods available, they are called enumerable methods such as #map, #select and #group_by. They are available to the Array and Hash classes via the Enumerable Module.

Ive decided to review #map because I've used it in several of my solutions this week. Review the Ruby documentation for map

The map method provides a powerful and consice way to re-build (re-map) an array based on certain conditions.

Using .map to re-build an array and perform logic on each iteration


array = [1,2,3,4]
array.map { |i| i * i }
#=> [1, 4, 9, 16]

Using .map in a chain of methods. Split a single number into an array and perform logic on each iteration


def number_split(number)

end

number_split(28434)
#=> ["Number 2", "Number 8", "Number 4", "Number 3", "Number 4"]

Yarrr, Arrays & Hashes

March 15th 2015 / Technical Blog

Arrays and hashes store collections of objects. Objects in the array can then be accessed in various ways depending on your requirements.

The items in an array or has can be strings, integers even another array. Pretty much anything which is an object can be stored in an array.

Arrays are always made of key:value pairs. In an Array the key is a sequentially assigned integer. Hashes allow the key to be defined by the user.

Let's look at some examples of creating, reading and accessing an Array

Arrays

Creating an Array You can create an array several ways.


# Create Array literal
array = []
# Create Array Explicit
array = Array.new

Unfortunately at this point I cannot tell if literal or explicit is better, or why you would use one over the other. My very basic understanding is that the literal invocation is simple a shortcut for the explicit. In practice, so far, I have just used the literal invocation.

Example Array with key:value pairs shown below

array = [1,2,3,"cat","dog","pig"]
# array keys [0,1,2,3,4,5]
# array values: [1,2,3,"cat","dog","pig"]

Note: In an array the index begins at 0.

Accessing an object in the array


array[0]
# Would access the Array index 0 (the key) and return an integer value of 1
array[3]
# Would access the Array index 3 (the key) and return an string value of "cat"

Hashes

Hashes are similar to arrays but as mentioned earlier you can define the key as well as the value. Perhaps this is useful when more semantic data storage and lookup is a requirement.

Create a new Hash


hash = Hash.new()

Example Hash

hash = {
    "Galaxy" => "cat"
    "Minty" => "dog"
    "Arty" => "dog"
}

Access a Hash


hash["Galaxy"] #=> cat
# Would access the Hash index "Galaxy" (the key) and return a string value of "cat"

CSS Positioning

March 9th 2015 / Technical Blog

Relative, absolute and fixed positioning are probably one of the more difficult concepts to grasp in CSS, so here's my take on things.

Typical code to position some using absolute positioning might look like this.


div {
    position: absolute;
    top: 300px;
    left: 300px;
}

Divs by default are display: block; and position: static; they will follow the document flow and each form their own line. In general think of all elements as static; p tags or h1 tags etc (thou they do have different default display values

position: relative; is the first level of positioning. Applying position: relative; to an element allows you to position it using top, right, left, bottom. For example top: 40px;. It also enables z-index, which allows you to control the stacking order of the element.

position: absolute; has the same qualities of position relative with one key difference. It takes it's position from the nearest parent in the DOM with position relative. This is often desirable as it can give you more accurate control when positioning a nested div's deep within a wrapper.

position: fixed; positions the element relative to the viewport (the edge of the page) and It remains "fixed" when you scroll. It has limited utility and is generally used for fixed navigation bars at the top of a page (you know when you scroll the menu stays on top)

Position is powerful, and is useful because it enables z-index and allows you to stack. But otherwise it should be used with caution and is not a replacement for more core positioning properties like margin, padding, display and float

When should I Use Position?

You should not need to declare a position value or use top, right, bottom, left for general layout. Margin, padding, float and display are the correct tools for structutural layout. This is because they do not disrupt the document flow.

Positioning is normally used when you need to use stacking. Positioning enables the z-index. So if you need to stack a div something under or over another div then it's time for positioning

Document Flow

The biggest weakness and strength of position (besides z-index) is how it removes the element from the normal document flow. Remember that when you enable position relative and set a position or when you enable position absolute (with or without a position) you remove the element from the document flow. This means it wont "push" other content down the page. That normally breaks layouts, especially if the content is dynamic or via a CMS.

Handle With Care

Absolute positioning in particular should be handled with care. Always take a moment to consider if absolute positioning is really required. Could you change the design? Can you re-factor the markup so you can use margin and float? Can you use background images? etc

Like a Version

February 27th 2015 / Technical Blog

Think about some of the greatest musicians of all time. Bob Dylan, Bob Marley, Deadmau5 (to be decided). What have they got in common? Their songs have all been covered and remixed dozens of times. People take their work and create derivitave works, sometimes better, sometimes worse.

Programmers and developers do the same thing. They work on each others code. They modify, expand and contribute their own thoughts to the original source.

To do this they use version control. It allows everyone to have a copy of the original work and make changes. Better yet, a log is kept of all the previous versions.

So with version control you can listen to original version of "all along the watchtower" by Bob Dylan and compare it to the far superior version by Jimi Hendrix. You can learn from both songs, can debate which is best, and ultimately decide to listen to the version we prefer.

There are 2 popular version control systems today: SVN and git. SVN is mainly used by old people who work for Microsoft or program java. git is used by everyone else.

Those who prefer git almost always use Github. They are not the same thing, but they compliment each other. Github allows people to share their version controled code over the web with ease. It acts as a server, offers an extended set of tools and has an elegant UI which makes the process of version control a hell of a lot easier then git alone provides.

So if your going to build something or code something and you want other people to listen, give feedback and riff of your original work your going to need version control. And the popular choice is git combined with Github.
